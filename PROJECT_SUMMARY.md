# 学员报名系统开发总结报告

**日期**：2026-01-27  
**项目**：学员报名管理系统 (Student Enrollment System)  
**技术栈**：React + Vite + TailwindCSS (前端), Supabase (后端/数据库/Auth/Storage)

---

## 1. 项目概况
本项目旨在构建一个多租户的学员报名管理系统，支持移动端访问。系统包含两个核心角色：
- **普通管理员**：管理自建班级和学员，生成报名二维码，导出学员数据。
- **超级管理员**：全局上帝视角，具备跨租户数据查询、僵尸账号清理、数据救援及强制级联删除权限。

---

## 2. 开发历程与关键需求实现

### 阶段一：基础架构与身份识别
**核心功能**：
1.  搭建管理员/学员双端登录系统。
2.  **控制台身份显示**：顶部栏实时显示当前登录的管理员姓名和账号。
3.  **移动端适配**：修复了 CSS `hidden lg:flex` 导致小屏幕无法显示用户信息的问题，确保移动端体验一致性。

### 阶段二：超级管理员权限与数据隔离 (RLS)
**核心功能**：
1.  **严格的数据隔离**：通过 Supabase RLS (Row Level Security) 策略，确保普通管理员只能访问自己的数据。
2.  **上帝视角**：超级管理员 (`admin_type = 'super'`) 可绕过 RLS 限制查看所有班级。
3.  **ID 映射修复**：解决了 `public.admins` 主键与 `auth.users` UUID 不一致导致的数据关联失效问题，通过后端 RPC 函数 `get_admin_class_stats` 实现精准统计。

### 阶段三：管理员删除与数据安全
**核心功能**：
1.  **级联删除机制**：创建 `SECURITY DEFINER` 级别的 RPC 函数，在删除管理员时自动清理关联的班级、学员及报名记录，防止残留孤儿数据。
2.  **权限突破**：利用后端函数绕过客户端权限限制，允许超级管理员直接删除 Auth 用户。

### 阶段四：数据救援（“幽灵账号”处理）
**核心功能**：
1.  **遗留数据扫描**：开发探测器识别无主（admin_id 指向不存在用户或为 NULL）的班级数据。
2.  **所有权转移**：实现了将无主班级一键转移给现有管理员的功能，增强了系统的容错性。

### 阶段五：移动端兼容性攻坚
**问题**：部分 Android/iOS 设备在使用 `browser-image-compression` 库压缩图片时导致浏览器崩溃或卡死。
**解决方案**：
1.  **弃用第三方库**：移除重型压缩库。
2.  **原生 Canvas 重构**：编写轻量级的 `compressImage` 工具函数，利用 HTML5 Canvas API 原生实现图片压缩与格式转换。
3.  **成效**：彻底解决了低端机型的内存溢出问题，提升了上传速度。

### 阶段六：高级学员管理与僵尸号清理
**需求**：
1.  **全局学员查询**：超级管理员需要跨班级、跨管理员检索学员。
2.  **僵尸号治理**：识别注册了账号但未填写任何资料的“空壳用户”。
**实现**：
1.  **RPC 函数封装**：开发 `get_all_students_overview` 函数，高效聚合学员、班级、管理员三方数据。
2.  **批量清理**：实现了对僵尸账号的批量筛选与一键物理删除。

### 阶段七：安全交互与体验优化
**核心功能**：
1.  **红色警报机制**：
    - 开发 `DangerDeleteModal` 组件，引入震动动画和红色主题。
    - **强制确认**：删除包含重要资料的学员时，强制要求管理员手动输入“确认删除”四个字，防止误操作。
2.  **文案与引导优化**：
    - 登录页增加“手机端建议使用 Chrome”的提示。
    - 将“分享二维码”更名为更直观的“班级报名入口”。
3.  **PDF 自动合并**：学员报名时，身份证正反面自动合并为单页 PDF，方便管理员归档。

## 3. 技术总结与最佳实践

1.  **Supabase RPC (远程过程调用) 的深度应用**：
    - 本项目大量使用 Postgres Function (PL/pgSQL) 处理业务逻辑。
    - **优势**：
        - **原子性**：级联删除等操作在数据库层作为一个事务执行，要么全成要么全败。
        - **权限控制**：通过 `SECURITY DEFINER` 让普通前端代码安全地执行高权限操作。
        - **性能**：减少了前端与数据库的多次往返交互。

2.  **前端性能优化**：
    - 图片处理完全在客户端（浏览器）完成，大幅减少了服务器带宽压力和存储成本。
    - 使用原生 API 替代重型依赖，减小了构建包体积。

3.  **防御性编程**：
    - 针对数据删除操作设计了“三级防护”：前端弹窗 -> 强制输入确认 -> 数据库层外键约束。
    - 处理了 `NULL` ID、遗留数据等多种边界情况。

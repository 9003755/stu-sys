# 上传功能修复复盘

## 文档目的

记录本次“手机浏览器上传身份证与一寸照失败”的完整排查、试错、定位与修复过程，沉淀可复用的诊断方法和改造经验，供后续移动端上传、页面恢复、浏览器兼容性开发参考。

## 问题背景

项目为学员报名系统，学员在手机扫码后进入报名页，需要上传：

- 身份证正面
- 身份证反面
- 一寸白底证件照

初始现象：

- 安卓手机使用 Chrome 可以正常上传
- 三星浏览器、微信内置浏览器上传失败
- 用户从相册选择照片，点击“确定”后，页面先出现“加载中...”，随后回到上传前状态
- 图片没有进入“上传中”状态，也没有出现预览图

用户已明确反馈：

- 问题与图片大小、类型无关
- 三个上传项都失败
- 微信和三星浏览器表现一致

## 初期假设与试错

### 假设一：前端压缩链路兼容性差

初期判断上传失败可能来自：

- `FileReader`
- `Image`
- `canvas.toBlob`
- `new File`

因为这套链路在部分移动浏览器中确实容易出兼容问题。

采取过的尝试：

- 压缩失败时回退原图上传
- 使用 `URL.createObjectURL` 替代 base64 处理
- 降低压缩过程的内存消耗
- 放宽图片格式支持

结果：

- 未解决问题
- 说明问题并不只在图片处理函数本身

### 假设二：页面过重导致移动端返回后崩溃

随后做了页面减重尝试：

- 路由懒加载
- 地址大数据按需加载
- 降低 `StudentProfile` 首屏负担

结果：

- 现象仍存在
- 问题被缩小到“选图返回后的页面生命周期”

## 关键调研

### 外部案例调研结论

对安卓浏览器、WebView、微信内置浏览器、三星浏览器的公开案例进行调研后，得到以下判断：

- 安卓底层文件上传能力本身成熟，不存在“安卓不会上传文件”的问题
- 差异重点不在上传 API，而在浏览器宿主如何处理：
  - `input[type=file]`
  - 系统相册切换
  - 返回页面时的生命周期恢复
- Chrome 通常更稳定，是因为它更容易保住原页面实例与 JS 上下文
- 微信内置浏览器、三星浏览器更容易在“选图返回”时重建页面

### 竞品网站对比调研

对一个可在微信中正常上传身份证的网站进行了技术识别，确认：

- 页面技术栈偏 `Vue`
- 上传组件为 `Element` 体系
- 上传容器类名为 `el-upload`
- 加载了微信 JS-SDK，但上传行为更像标准原生文件选择器，而非主要依赖 `wx.chooseImage`

得到的关键启发：

- 对方成功的关键不在“用了微信黑科技”
- 更可能在于：
  - 上传组件更成熟
  - 上传区更独立
  - 页面恢复链路更平滑
  - 选图返回后没有整页重新走一遍认证与路由链路

## 关键证据采集

### 第一次关键埋点

为了判断问题到底出在：

- 文件选择回调没回来
- 页面被浏览器重建
- 上传接口失败

在 `StudentProfile` 中加入了最小化调试埋点，记录：

- `picker-click`
- `change-file`
- `upload-start`
- `upload-success`
- `upload-error`
- `visibilitychange`
- `pagehide`
- `pageshow`
- `focus`
- `mount`
- `unmount`

### 三星浏览器日志证据

三星浏览器失败时拿到的关键日志表现为：

- 有 `picker-click`
- 有 `visibilitychange hidden`
- 有 `visibilitychange visible`
- 有 `focus`
- 随后出现 `unmount`
- 然后出现新的 `mount`
- 没有 `change-file`
- 没有 `upload-start`

这份日志非常关键，证明：

- 不是上传接口失败
- 不是压缩函数执行失败
- 不是文件选中了但上传报错
- 而是页面在相册返回后被卸载再重挂
- 文件选择结果没有成功回到原来的页面实例

## 根因定位

继续检查本项目代码链路后，确认不仅是上传组件问题，更是上层页面恢复问题：

### 根因一：AuthProvider 在恢复会话前直接不渲染子树

原逻辑中：

- `AuthContext` 启动后先 `getSession`
- 在 `loading` 为 `true` 时，直接不渲染 children

这意味着：

- 一旦浏览器在选图返回时触发页面恢复
- 上传页会先被上层认证恢复逻辑整体掐掉

### 根因二：关键页面采用 lazy + Suspense

原逻辑中：

- `StudentProfile`
- `EnrollConfirmation`

都处于懒加载链路中。

相册返回后如果页面恢复重新走路由：

- 会先掉进全屏 `Suspense fallback`
- 用户就会看到“加载中...”

### 根因三：报名页本身还会重新检查状态

`EnrollConfirmation` 在恢复后会重新：

- 查询班级
- 查询 profile
- 查询 enrollment

这会进一步拉长页面恢复链路，并增加重新显示上传表单前的等待与重建概率。

## 最终修复策略

本次最终有效修复不是单点修改，而是三层联动。

### 一层：保留上传区的独立上传能力

保留并优化上传组件本身：

- 选图后独立进入上传流
- 成功后只把 URL 回填给表单
- 上传区自身维护预览和错误状态

这样主表单只依赖上传结果，不强耦合临时文件对象。

### 二层：认证恢复改为尽量平滑

在 `supabase.js` 中：

- 给学生端会话增加显式 `storageKey`
- 增加同步读取已存用户能力

在 `AuthContext.jsx` 中：

- 优先从 `sessionStorage` 同步恢复用户
- 不再因 `loading` 直接隐藏整棵子树

作用：

- 页面恢复时尽量保住现有 React 树
- 降低“认证恢复导致上传页被掐掉”的概率

### 三层：关键报名页不再走容易闪断的恢复链路

在 `App.jsx` 中：

- 将 `StudentProfile`
- `EnrollConfirmation`

从懒加载改为直接加载。

作用：

- 相册返回后，尽量避免页面先掉进全屏 `加载中...`

在 `EnrollConfirmation.jsx` 中：

- 加入报名页状态缓存
- 缓存 `classInfo`
- 缓存 `enrollStatus`
- 缓存 `profileId`

作用：

- 页面恢复时优先回到资料填写态
- 减少重新查询后再决定渲染哪个页面的延迟与抖动

## 修复后的验证结果

微信内置浏览器成功日志：

- `picker-click`
- `visibilitychange hidden`
- `visibilitychange visible`
- `change-file`
- `upload-start`
- `upload-success`

并且：

- 全程 `pageInstanceId` 没变化
- 没有 `unmount -> mount`

这说明：

- 文件选择回调已成功返回当前页面实例
- 页面恢复链路已被稳定住
- 上传流程完整闭环

## 最终结论

这次问题的本质不是“某些浏览器不支持上传图片”，而是：

- 在特定移动浏览器中
- 选图返回后页面恢复链路不稳定
- 项目原有上层认证与懒加载机制放大了这个问题
- 最终表现成上传前页面被重挂，导致文件对象丢失

最终有效修复方向不是继续微调压缩参数，而是：

- 稳住页面恢复链路
- 减少恢复期整页重挂
- 让上传区和主表单解耦

## 本次经验总结

### 经验一：移动端上传失败，先分层判断

不要一开始就假设是“上传接口有问题”，应优先区分：

- 文件选择回调是否触发
- 页面是否被重建
- 上传是否真正开始
- 上传请求是否失败

### 经验二：日志要覆盖生命周期，不只覆盖业务函数

针对移动端浏览器问题，必须采集：

- `visibilitychange`
- `pagehide`
- `pageshow`
- `focus`
- `mount`
- `unmount`

否则无法判断是业务逻辑错，还是浏览器宿主行为导致状态丢失。

### 经验三：不要只改上传函数，要看页面宿主链路

如果用户表现为：

- 选图后出现全屏“加载中...”
- 返回上传前状态

优先怀疑：

- 路由懒加载
- 认证恢复
- 上层页面重挂

而不是只盯着 `input[type=file]`。

### 经验四：竞品分析要抓“页面恢复策略”，不要只看控件样式

竞品网站虽然表现为“微信里能传”，但深挖后发现关键不一定是：

- 微信专属上传 API

更可能是：

- 成熟的上传组件
- 轻量独立的上传区
- 更稳定的页面恢复链路

### 经验五：调试能力应可保留，但正式环境默认隐藏

本次埋点是定位问题的关键工具，应保留：

- 本地日志采集
- 生命周期日志
- 上传步骤日志

但正式环境中应默认隐藏调试面板，只在开发环境或指定参数下开启。

## 后续建议

### 建议一：继续保留上传诊断能力

建议长期保留日志采集函数，但将面板入口默认关闭，仅在以下场景启用：

- 开发环境
- `?uploadDebug=1`
- 特定测试账号

### 建议二：新增移动端回归清单

后续每次涉及报名页、认证、路由、上传组件改动后，至少验证：

- 安卓 Chrome
- 微信内置浏览器
- 三星浏览器

### 建议三：优先避免关键报名页的大规模恢复闪断

后续如果继续优化包体或路由，不要轻易把报名关键页重新放回高风险懒加载链路，除非已有充分验证。

### 建议四：若后续再遇到上传异常，优先复用本次方法

推荐标准排查顺序：

1. 确认是否出现 `picker-click`
2. 确认是否出现 `change-file`
3. 确认是否出现 `unmount -> mount`
4. 确认是否出现 `upload-start`
5. 确认是否出现 `upload-success`
